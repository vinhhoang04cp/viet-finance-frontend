## 0. Authentication & multi-tenant (quick reference)

> Full prose in `CLAUDE.md` → "Authentication & multi-tenant session". This is the gotcha list.

- **Token split:** access JWT in `localStorage` (`lib/auth/storage`), refresh token in an **httpOnly
  cookie set by the backend** — `refreshTokens()` sends **no body**, just `credentials:"same-origin"`.
  Never try to read/store the refresh token in JS.
- **`apiFetch` is the only authed fetch** (Invoice/Revenue/Tax/Users/Tenants). On 401 it refreshes
  **once** (concurrent 401s queued behind a single `/auth/refresh`), retries, and on failure emits the
  `vf:auth-expired` window event → `AuthProvider` resets + redirects to `/auth/login?returnUrl=…`.
  `authApi.ts` uses raw `fetch` (no Bearer/refresh) — don't route it through `apiFetch`.
- **Session = `AuthProvider` Context** (`useAuth()`): `user`, `tenants`, `currentTenant`, `role`,
  `canWrite` (OWNER/ACCOUNTANT; VIEWER read-only), plus actions. **Temp token** (no `tid`) is held in a
  ref for `selectTenant`/`createTenant` after multi-tenant login or OAuth `noTenant`.
- **`decodeJwt` does NOT verify** — client-side base64 decode only, to read `tid`/`role`/`exp`. The
  backend is the verifier.
- **Route gating is client-side** (`AppShell` + `RequireAuth`) because the token is in localStorage,
  invisible to Next middleware. `/auth/*` public; `/onboarding` authed-no-tenant; rest authed+tenant.
- **OAuth `loginWithGoogle` bypasses the proxy** — it navigates to `NEXT_PUBLIC_BACKEND_URL` directly
  (the proxy can't carry the OAuth redirect/session); the backend returns to `/auth/callback?code=…`.
- **`switchTenant` hard-reloads** `/` (with a sessionStorage flash toast) — there's no global data cache
  to invalidate, so a full reload is the simplest way to drop the previous tenant's data.

## 1. Localization (Vietnamese UI)

The accountant-facing UI is being localized to **pure Vietnamese**. There is **no i18n framework** —
strings are hardcoded directly in the JSX. Each localization pass is a direct text swap.

### The one hard rule: display text ONLY
Translate **only** what the user sees: JSX text nodes, `label=` / `placeholder=` / `aria-label=`
attribute values, section headings, button labels, and user-facing error/validation message strings.

**Never** change anything that is data or logic — doing so breaks state binding or the API contract:
- `id="…"`, `value={form.…}`, `onChange={set("…")}`, and the form-state keys themselves
- component prop names, `types/*` field names
- API values. The approval **status** is the canonical trap: it stays the raw backend string
  (`PENDING` / `APPROVED` / `REJECTED`). `StatusBadge` renders a Vietnamese label via `statusLabel()`
  for **display only**; `statusVariant()` (badge colour) and the value sent to `PATCH /{id}/status`
  both keep the raw English value.

### Shared-component caveat
`components/upload/ScanModeTabs`, `FileDropzone`, and `BatchUpload` render on **both** scan pages
(`invoices/scan` and `revenues/scan`). Translating them localizes both at once — so localize both scan
pages together, or one page ends up with Vietnamese upload chrome under English headings.

### Backend-originated strings are NOT localized client-side
`describeError()` surfaces the backend `ErrorResponse.message`, and `BatchAcceptedResponse.message`
(rendered in `BatchUpload`'s success card next to the translated static text) come from the server in
English. These are left as-is; localizing them would require backend i18n.

### Status: done vs. remaining
- **Localized:** `RevenueReviewDialog`, `RevenueTable`, `revenues/page.tsx` (dashboard),
  `invoices/scan/page.tsx`, `revenues/scan/page.tsx`, `upload/ScanModeTabs`, `upload/FileDropzone`,
  `upload/BatchUpload`, `StatusBadge`. **All newer surfaces are Vietnamese-native:** the auth pages
  (`auth/{login,register,callback}`), `onboarding`, `settings/*`, the `/dashboard` + `/tax` analytics
  pages, `Navbar`, and the `components/statistics/*` + `components/auth/*` components.
- **Still English (legacy invoice surface):** `app/page.tsx` (invoice home / review queue + KPI cards),
  the invoice table, and the **Invoices/Revenue nav groups in `Sidebar.tsx`** ("Approval queue", "Scan
  invoice", "Reports", "Scan report") — note the newer "Tổng quan" group (Thống kê / Thuế GTGT) is
  already Vietnamese, so the sidebar is now mixed.

## 2. Analytics dashboards: monthly timeline + drill-down

> Full prose in `CLAUDE.md` → "Tax & statistics dashboards". This is the gotcha list.

- **Two different "monthly" shapes — don't confuse them.** The OLD `/statistics/monthly` is **12 buckets
  within one year** (`MonthlyTaxPoint`, `month` only; "all years" collapses calendar months across years).
  The dashboards now use the **timeline** (`TimelinePoint` with `year`+`month`, `GET …/statistics/timeline`
  or `/tax/calculation/timeline`) = a CONTINUOUS rolling window that spans year boundaries. The old
  `MonthlyTaxStatistics`/`emptyMonthly`/`mergeMonthly`/`sumMonthly` helpers still exist but are no longer
  wired into the pages — use `recentRange`/`mergeTimeline`/`sumTimeline`/`timelineShort`/`timelineLong`/
  `monthRangeOf` + `TIMELINE_RANGES` from `lib/statistics.ts`.
- **Charts are clickable.** `MonthlyTaxBarChart` / `MonthlyVatBarChart` take `onSelectMonth`; the click is
  wired on each `<Bar onClick={(_, index) => ...}>` (recharts gives the category index → map back to the
  data point). `MonthlyVatBarChart` derives its X label from each point's `periodStart` (`MM/YY`), NOT a
  fixed 12-month array, so it renders a continuous axis.
- **Drill-down fetch is a second effect keyed on the selected month** (+ module/status). It sets
  `detailLoading` in the click handler (not synchronously in the effect — the repo's
  `react-hooks/set-state-in-effect` rule). Dashboard derives the month range with `monthRangeOf(point)`;
  the tax page reuses the point's own `periodStart`/`periodEnd`. Dashboard fetches per **module**
  (invoice-only / revenue-only / both); tax always fetches both sides via `fetchVatDetails`.
- **Backend caps the window** (`MonthRange.MAX_MONTHS = 120`) and defaults a null bound to the last 12
  months — so even a hand-crafted huge `from` won't fan out. The frontend only ever sends 6/12/24-month
  ranges from `recentRange`.
